# INSTITUTION ROOM MANAGEMENT SYSTEM PROJECT
This is our Software Engineering Lab Project for the 6th Semester. 
## OUR PROBLEM STATEMENT 

An everyday problem considering our institution, where there is a frequent clash in the rooms allocated to different batches and faculties. And everyone has to do the hectic task of finding the free(empty) room. Hoping no other batch comes and claims that they have an allocated class there, wasting both time and labour for it. Even if an anonymous class is selected on finding the allocated room occupied, the same problem is faced by the batch to which that room is specifically allocated.

## STAKEHOLDERS
The main stakeholders that our application will focus on are the institution’s management (primary stakeholders)  and the people using the infrastructure of that institution (primary stakeholders). 

_Institution management_: The management need not pre allocate the rooms and can monitor the list of  allocated rooms in real time.

_Faculty_: The  faculty will be involved in allocation and deallocation of the rooms in which he/ she is interested to take the class based on the room availability.

_Students_: Students will easily get information through the application regarding their allocated rooms for respective classes, and some free rooms if they require so.

## SOFTWARE MODEL USED
The process model which we propose to proceed our development with, is **_SCRUM model_**, as it offers flexibility and incremental/ rapid development.

_Project Planning_:
 As we have previously mentioned we have three different sets of users / stakeholders we can use the flexibility and incremental development approach offered by the scrum model to divide the different needs in smaller chunks for development. In this phase we can inquire all three sets of users for their needed features in the software. We can get a good grasp of the needs of the faculty and institution administration and the students.co
 
_Incremental development_:
	In this stage we are dividing the needs of the group(s) of users, so that the scrum model can be
	implemented easily. With this step, we can develop the software for a set of users and then proceed 
	for the development of the next set of users.
 
_Review and feedback_:
With each sprint, the product evolves incrementally, with the new features being added and the existing features refined based on the feedback of the client/ user. Moreover the team reflects on the sprint process, discussing the all round development, any required improvements, or adjustments required for the next sprint. Accordingly, actionable items are identified to address areas of improvement of the application.
